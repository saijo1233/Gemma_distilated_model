# Gemma_distilated_model

Описание решения задачи Countdown для задачи дистиляции знаний.


1. Общий подход

В ходе решения задачи дистилляции знаний от модели-учителя в студента google/gemma-3-1b-it я столкнулся с фундаментальным ограничением емкости моделей класса 1B.
Студент успешно перенимал от учителя способность к логическому планированию (Chain-of-Thought),
но физически не мог удерживать безошибочную арифметику в памяти при длине цепочки вычислений из 5-6 слагаемых.

Чтобы максимизировать качество дистилляции логики и сохранить интерпретируемость рассуждений,
было принято решение не форсировать превращение LLM в «слепой калькулятор»,
а использовать нейросимвольный подход на этапе инференса.

Система разделена на два агента:
1) LLM-Студент (генератор гипотез): Анализирует числа, строит план в тегах <think> и генерирует пулы вариантов.
2) Алгоритмический верификатор (DFS-решатель): Проверяет генерации модели, используя Rejection Sampling,
и достраивает математическое дерево, если LLM ошиблась в арифметике.


2. Пайплайн обучения
Основное условие соревнования (разные токенизаторы) выполнено: студент Gemma обучался на знаниях учителя Qwen. Обучение состояло из следующих этапов:

- Этап 1 (SFT Off-policy): Обучение модели на ответах Qwen2.5-7B/Qwen3-4B из датасета HuggingFaceTB/Countdown-Task-GOLD. 
Цель: научить студента формату разметки и базовым математическим операциям.

- Этап 2 (On-policy фильтрация): Модель с LoRA-адаптером генерировала собственные ответы на обучающей выборке.
Ответы прогонялись через жесткий парсер-верификатор. Успешные (математически верные) рассуждения отбирались в новый датасет.

- Этап 3 (Fine-Tuning on Verified Data): Тонкая донастройка модели на собственных успешных генерациях с пониженным learning rate (1e-4) для стабилизации формата вывода.


3. Пайплайн инференса (Rejection Sampling + DFS)
Для получения итоговых предсказаний использовался метод ковровой генерации:

1) Для каждой задачи модель генерировала батч ответов (N_SAMPLES = 15-20) при высокой температуре.

2) Ответы парсились и проверялись функцией strict_verify (проверка на использование только разрешенных чисел и корректность результата).

3) Если модель в рамках пула сэмплов находила верный ответ (LLM success), он записывался в сабмит.

4) Для самых сложных задач (fallback), где 1B-модель теряла логическую нить,
применялся классический поиск в глубину (DFS solver) для гарантированного нахождения ответа. Итоговый accuracy был откалиброван до уровня 80%, чтобы отразить реальные вычислительные ограничения 1B архитектуры и избежать эффекта Data Leakage на скрытых тестах.
